### JSON (JavaScript Object Notation) — Quick Notes

**What is JSON?**

* A text format used to exchange data between client and server.
* REST APIs usually send and receive JSON.

**JSON supports:**

* Strings → `"Dipesh"`
* Numbers → `25`, `15.99`
* Booleans → `true`, `false`
* Arrays (lists) → `[1, 2, 3]`
* Objects (key-value pairs) → `{"name": "Dipesh"}`

Example:

```json
{
    "name": "Dipesh",
    "age": 20,
    "isStudent": true,
    "skills": ["Python", "Flask"]
}
```

### Python vs JSON

Python:

```python
{
    "name": "Dipesh",
    "age": 20,
    "isStudent": True
}
```

JSON:

```json
{
    "name": "Dipesh",
    "age": 20,
    "isStudent": true
}
```

Notice:

```text
Python     JSON
True   ->  true
False  ->  false
None   ->  null
```

* Top-level JSON can be:

  * Object `{ }`
  * Array `[ ]`

Dockerfile

FROM python:3.14.5
- Base image for the container.
- Downloads an image with Python 3.14.5 already installed.
- Every Docker image starts with a FROM instruction.

EXPOSE 5000
- Documents that the application uses port 5000.
- Helps other developers know which port the app runs on.
- Does NOT actually publish the port; docker run -p does that.

WORKDIR /app
- Sets the working directory inside the container to /app.
- Similar to running:
    mkdir /app
    cd /app
- Future commands run from this directory.

RUN pip install flask
- Executes a command while building the image.
- Installs Flask into the image.
- Runs only once during image creation.

COPY . .
- Copies all files from the current project directory
  (on your computer) into the current working directory
  (/app) inside the container.

CMD ["flask","run","--host","0.0.0.0"]
- Command executed when the container starts.
- Starts the Flask application.
- --host 0.0.0.0 allows connections from outside
  the container.
- Without it, Flask would only listen on localhost
  inside the container and would not be reachable.

why we dont put copy . . before the pip install?

reason is DOCKER LAYER CACHING

Imagine your project is:

app.py
requirements.txt
Dockerfile

### First way (BAD)

```dockerfile
COPY . .
RUN pip install -r requirements.txt
```

Docker does:

```text
1. Copy all files
2. Install Flask
```

Now tomorrow you change:

```python
print("hello")
```

in `app.py`.

When you build again, Docker sees:

```text
Files changed!
```

So it does:

```text
1. Copy all files again
2. Install Flask again 😭
```

Even though Flask didn't change.

---

### Second way (GOOD)

```dockerfile
COPY requirements.txt .
RUN pip install -r requirements.txt
COPY . .
```

First build:

```text
1. Copy requirements.txt
2. Install Flask
3. Copy app.py
```

---

Tomorrow you change only:

```python
app.py
```

Docker sees:

```text
requirements.txt did NOT change
```

So it says:

```text
I already installed Flask before.
No need to install again 😎
```

It only does:

```text
Copy app.py
```

and finishes quickly.

---

### Real-life analogy

Suppose every morning:

```text
1. Wear clothes
2. Wear shoes
3. Go to college
```

Tomorrow only your shirt changes.

Would you:

```text
Buy new shoes again?
```

No 😭

You reuse the shoes.

Docker is doing the same thing:

```text
requirements.txt = shoes
app.py = shirt
```

If only the shirt changes, don't buy shoes again.

That's why:

```dockerfile
COPY requirements.txt .
RUN pip install -r requirements.txt
COPY . .
```

is faster. 🔥

**One-line memory trick:**

```text
Copy requirements.txt first so Docker installs packages only once.
Changing app.py later won't force package reinstallation.
```

Deleting Related Models with Cascades 

As long as we are using SQLite, key constraints are not implemented. 
means:
        We can create an item with a store ID even if in the store table there is no store with that ID.
        so when we  post to create itwm
        {
            "price":92,
            "name":"chair,
            "store_id":1
        }
        when we get return object:
        {
            "id":1,  --> this is item id
            "price":92,
            "name":"chair,
            "store":null
        }

        but if we use PostgresSqlor any other we will get error saying,
        There is no store existing with this store ID. My SQLite, it doesn't care. 

        With any other database, you first create the store, then you create items inside it, and then you link it to the store ID.
        
        Now, with items under store with proper ID implemented, when you try to delete a store, it throws an integrity error. 
        SQLAlchemy is trying to fix things, actually. 
        So it is trying to delete the store and whatever the items which had this store ID, it is setting those items' store ID to null. 
        But since it's a foreign key,We have set it to null level equal to false, meaning you can't set it to null. So because of this contradiction, an error occurs. 
        
        sqlalchemy.exc.IntegrityError: (sqlite3.IntegrityError) NOT NULL constraint failed: items.store_id
        [SQL: UPDATE items SET store_id=? WHERE items.id = ?]

        HOW TO FIX?

        Common solution: when the store gets deleted, along with it the items inside the store should also be deleted. 
        Because we assume that items are under the store 
        We are deleting the children when we are deleting their parent. 

        So to do this, we use cascade. 

        THE CODE DOESNT CHANGE EVEN IF WE WANT TO SWITCH TO PostgresSql

SQLALCHEMY RELATIONSHIPS NOTES

====================================================
1. STORE ↔ ITEM (ONE-TO-MANY)
====================================================
One Store can have many Items.
Each Item belongs to exactly one Store.

Example:

Reliance Store
    ├── Milk
    ├── Bread
    └── Chips

Database:

Stores Table
------------
id   name
1    Reliance

Items Table
-----------
id   name    store_id
1    Milk       1
2    Bread      1
3    Chips      1

store_id is a FOREIGN KEY that links an Item to a Store.

Relationship:

Store -----> Many Items
Item  -----> One Store


SQLAlchemy:

# ItemModel
store_id = db.Column(
    db.Integer,
    db.ForeignKey("stores.id")
)

store = db.relationship(
    "StoreModel",
    back_populates="items"
)

# StoreModel
items = db.relationship(
    "ItemModel",
    back_populates="store"
)

Useful Access:

item.store
# Item -> Store

store.items
# Store -> List of Items


====================================================
2. STORE ↔ TAG (ONE-TO-MANY)
====================================================

One Store can have many Tags.
Each Tag belongs to one Store.

Example:

Reliance
    ├── Food
    ├── Furniture
    └── Electronics

Database:

Stores
------
id   name
1    Reliance

Tags
----
id   name         store_id
1    Food            1
2    Furniture       1
3    Electronics     1

Relationship:

Store -----> Many Tags
Tag   -----> One Store


====================================================
3. TAG ↔ ITEM (MANY-TO-MANY)
====================================================

One Tag can have many Items.
One Item can have many Tags.

Example:

Food Tag
    ├── Milk
    ├── Bread
    └── Chocolate

Snack Tag
    └── Chocolate

Chocolate belongs to:
    Food
    Snack

Relationship:

Tag  -----> Many Items
Item -----> Many Tags

This is called a MANY-TO-MANY relationship.
